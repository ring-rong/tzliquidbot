## 1. КРИТИЧЕСКИ ВАЖНЫЕ ПРАВИЛА (не обсуждаются)

1. **Бот запускается в Docker.** Финальная проверка = \`docker compose up --build\` (или
   \`docker build && docker run\`) поднимает бота без ручных доработок. Никакого
   "у меня локально работает" — только контейнер.
2. **Модульность обязательна.** Ни одного файла-простыни на 400+ строк со смешанной
   логикой (хендлеры + валидация + HTTP-клиент + тексты в одном месте). Структура ниже
   (раздел 4) — обязательный минимум, реорганизацию можно углублять, но не упрощать.
3. **Ноль явного "мусора от ИИ".** Это отдельный жёсткий критерий, см. раздел 8.
   Ревьюер увидит закомментированный код, "TODO: implement later", избыточные
   docstring'и уровня "This function returns the name" — и это будет сразу видно, что
   код не вычитан.
4. **Бот вежливый.** Все пользовательские сообщения — на "Вы", без канцелярита и без
   утечки технических деталей (ни одного traceback, ни одного "Error: ConnectionError"

5. **README.md подробный и честный.** Не "заглушка для галочки" — реальное описание
   стека с обоснованием выбора, структуры проекта, инструкции по запуску, настройке
   Mock-API и тестированию. См. раздел 10.
6. **Версии и API библиотек — не по памяти.** aiogram, pydantic и подобные библиотеки
   меняют мажорные API между версиями. Перед использованием конкретного синтаксиса
   (FSM, роутеры, фильтры) сверься минимум с двумя источниками: актуальный \`pyproject.toml\`/

   обязательный инструмент для этого, не опция). Не пиши код "как в aiogram 2.x", если

## 1.5. TOOL: CONTEXT7 MCP — ОБЯЗАТЕЛЬНОЕ ИСПОЛЬЗОВАНИЕ ДЛЯ ВНЕШНИХ БИБЛИОТЕК

Context7 — MCP-сервер, отдающий актуальную, версионированную документацию библиотек
прямо в контекст, вместо того чтобы полагаться на обучающие данные с датой отсечки.
Это прямая реализация правила 1.6: не доверять памяти в вопросах версий/API.

**Когда использовать (конкретные триггеры, не "когда покажется нужным"):**
- Перед тем как написать код, использующий что-то специфичное для конкретной
  библиотеки: роутеры/фильтры/FSM aiogram, \`field_validator\`/\`model_config\` Pydantic
  v2, \`BaseSettings\` из \`pydantic-settings\`, retry/timeout-конфигурацию \`httpx\`,
  моки \`respx\` в тестах.
- Перед тем как писать код "как я помню, это делается" для любой из этих библиотек —
  особенно там, где ты не уверен, v1-паттерн это или v2 (Pydantic это классическая
  ловушка: \`@validator\` vs \`@field_validator\`, \`class Config\` vs \`model_config\`).
- Если в процессе отладки код не работает так, как ожидалось, и есть подозрение, что
  синтаксис/API устарел — до попытки чинить наугад, свериться с докой.

**Когда НЕ использовать:**
- Для своего собственного кода проекта (\`validators/\`, \`handlers/\`, \`services/\`) —
  это не внешняя библиотека, Context7 тут не поможет и не нужен.
- Для стандартной библиотеки Python (\`asyncio\`, \`json\`, \`re\`, \`datetime\`) — она
  стабильна между минорными версиями, тратить вызов MCP нет смысла.
- Повторно для уже проверенной в этой же сессии версии той же библиотеки — резолвь
  один раз за задачу/фазу, дальше переиспользуй тот же \`context7CompatibleLibraryID\`,
  а не гоняй \`resolve-library-id\` на каждую мелкую подзадачу.

**Как использовать (точная последовательность вызовов):**
1. \`resolve-library-id\` с \`libraryName\` (например \`"aiogram"\`) → получаешь список
   кандидатов с ID вида \`/aiogram/aiogram\` и trust score. Если пользователь/CLAUDE.md
   уже даёт точный ID в формате \`/org/project\` или \`/org/project/version\` — этот шаг
   можно пропустить и сразу идти в шаг 2.
2. \`get-library-docs\` с \`context7CompatibleLibraryID\` из шага 1, опционально \`topic\`
   (например \`"FSM"\`, \`"routers"\`, \`"validators"\`) — узкий topic даёт релевантнее
   выдачу, чем общий запрос по всей библиотеке. \`tokens\` можно не указывать, дефолт
   обычно достаточен для точечного вопроса.
3. Не более ~3 вызовов \`get-library-docs\` на один вопрос/развилку — если после трёх
   попыток документация не даёт ответа, переходи ко второму источнику (см. failure
   mode ниже), не зацикливайся на MCP.

**Failure mode:**
- \`resolve-library-id\` вернул неоднозначные/нерелевантные кандидаты → не гадай, бери
  результат с максимальным trust score и явно упомяни выбор в \`DEVLOG.md\`, либо, если
  реально неоднозначно (несколько похожих библиотек под похожим именем) — уточни у
  владельца.
- \`get-library-docs\` вернул "Documentation not found"/пусто → почти всегда означает
  невалидный или устаревший \`context7CompatibleLibraryID\`, а не то, что доки не
  существует — перезапроси \`resolve-library-id\`, не изобретай ID вручную.
- Context7 недоступен (сетевая ошибка MCP, нет соединения) → второй источник из
  правила 1.6: реальный установленный пакет (\`pip show <pkg>\`, чтение исходников в
  site-packages) + официальный сайт/GitHub через \`web_search\`/\`web_fetch\`. Зафиксируй
  в \`DEVLOG.md\`, что Context7 был недоступен для этой проверки и чем заменили.

**Интеграция с остальным процессом:** каждая содержательная проверка через Context7,
которая на что-то повлияла (выбор паттерна, обнаружение, что API отличается от
ожидаемого) — это ровно тот тип решения, который требует записи в \`DEVLOG.md\`
(раздел 9): "почему aiogram FSM устроен так, а не иначе" — хороший пример записи,
для которой Context7 и есть источник ответа.

## 2. ТЕХ. ЗАДАНИЕ — СЖАТО (полный текст ТЗ у пользователя, сверяйся при сомнениях)

**User flow:**
\`/start\` → приветствие + предложение пройти опрос → запрос имени (2–50 символов) →
запрос телефона (валидация формата, напр. \`+7 (XXX) XXX-XX-XX\` и вариации) → выбор
времени связи кнопками (Утро / День / Вечер) → сводная карточка ("Проверьте ваши
данные: Имя, Телефон, Время") с кнопками **[Отправить заявку]** / **[Заполнить заново]**
→ по нажатию "Отправить" — асинхронный POST с JSON на внешний Mock-API → бот сообщает
результат.

**Mock-API (внешний, не пишем сами):** кандидат регистрирует эндпоинт в Beeceptor /
Webhook.site / аналоге (напр. \`https://<name>.free.beeceptor.com/v1/leads\`), настраивает
там: (а) задержку ответа 1.5–2 сек, (б) правило, при котором часть запросов (каждый
5-й или отдельный путь) отвечает 500 или 429.

**Retry-логика (обязательные числа, не менять):**
- Ловим сетевые ошибки и коды 500 / 429.
- Пользователю: «Сервер CRM временно занят, пытаюсь отправить еще раз...»
- Ждём 3 секунды, повторяем.
- Максимум 3 попытки. После третьей неудачи — фиксируем критический сбой и сообщаем
  пользователю (без пугающих формулировок, но честно — заявка не отправлена).

**Асинхронность — не опция, а требование:** пока один пользователь ждёт ответ от
Mock-API (1.5–2 сек, с ретраями — до ~9+ сек в худшем случае), бот обязан продолжать
обслуживать остальных без блокировки event loop. Любой блокирующий вызов
(\`requests\`, \`time.sleep\`, синхронный файловый I/O в хендлере) — критическая ошибка.

**Хранилище состояний:** любое лёгкое решение. По умолчанию используем встроенный FSM
aiogram (\`MemoryStorage\`) — этого достаточно для тестового задания; в README явно
написать, как переключиться на \`RedisStorage\` для мульти-инстанс продакшена, но
реализовывать Redis не обязательно, если не потребуется отдельно.

### 2.1. Mock-API — УЖЕ НАСТРОЕН И ПОДТВЕРЖДЁН РАБОЧИМ (владелец, вручную, до тебя)

Владелец задачи вручную создал endpoint на Beeceptor и настроил на нём weighted-response
правило (\`POST /v1/leads\`, 70% \`200\` / 20% \`500\` / 10% \`429\`, delay ~1.5–2 сек на успехе)
— именно то, что описано в \`API_CONTRACT.md\`. Это **не гипотеза и не TODO**, а
подтверждённый факт: владелец прогнал 5 запросов через \`curl\` и получил вживую
\`200, 200, 500, 500, 200\` — то есть правило реально матчится и отдаёт и успех, и ошибку.

- **URL (уже рабочий):** \`https://mpfbba1a2c2708d21084.free.beeceptor.com/v1/leads\`
- Это идёт в \`.env\` как \`CRM_ENDPOINT_URL=https://mpfbba1a2c2708d21084.free.beeceptor.com/v1/leads\`
  — см. правило 1.7: даже несмотря на то, что урл подтверждён и не секретен по своей
  природе (публичный бесплатный мок), он всё равно конфиг, а не хардкод в коде.
- **Из этого следует для workflow (раздел 5):**
  - Шаг 5.2 ("Mock-API — это ручной шаг владельца") закрыт. **Не спрашивай владельца,
    готов ли эндпоинт** — он готов, задавать этот вопрос заново не нужно.
  - Можно сразу переходить к end-to-end проверке (шаг 5.7) параллельно с разработкой,
    а не только в самом конце — реальный эндпоинт доступен с первого дня работы.
  - Не нужно объяснять в README, "как настроить Beeceptor" в сослагательном
    наклонении — можно писать по факту, что уже сделано, и даже вставить в README
    те самые проверочные \`curl\`-вызовы владельца как пример (раздел 10, п.7).
- Free-эндпоинты Beeceptor удаляются через 90 дней неактивности, а логи запросов
  чистятся через 15 дней — если бот не будет обращаться к нему какое-то время,
  проверь его доступность заново перед финальной сдачей, а не полагайся на память
  об этом факте из середины разработки.

## 3. РЕКОМЕНДУЕМЫЙ СТЕК (дефолт, можно аргументированно отклониться)

- **pydantic-settings** — типизированная конфигурация из \`.env\`.
- **Docker + docker compose** — обязательный способ запуска.
- Тесты — **pytest + pytest-asyncio + respx** (мок HTTP для детерминированной проверки
  retry-логики без завязки на реальный 5-й запрос к Beeceptor).

Если выбираешь другой стек (Node.js/Go/PHP, как разрешено в ТЗ) — сохрани все принципы
этого файла (асинхронность, модульность, retry-числа, вежливость, README), адаптировав
структуру раздела 4 под экосистему. **Не меняй стек посреди работы** — выбери один раз,
зафиксируй в README с обоснованием (раздел 10), дальше только следуй ему.

## 5. WORKFLOW (порядок работы)

1. **Настройка окружения.** Создать структуру проекта, \`requirements.txt\`/\`pyproject.toml\`

   500/429/latency живое и проверено владельцем через \`curl\`. Ничего не нужно
   регистрировать и не нужно ни о чём спрашивать — просто взять URL из раздела 2.1,
   положить в \`.env\` и спроектировать \`crm_client.py\` так, чтобы он читал его оттуда
   (не хардкодить). В README (раздел 10, п.6) описать это как готовый факт с
   подтверждающими \`curl\`-примерами, а не как инструкцию "как настроить" в будущем времени.
3. **Модели и валидация** (\`models/lead.py\`, \`validators/\`) — сначала, без Telegram —
   их проще всего протестировать pytest'ом изолированно.
4. **FSM и хендлеры** (\`states/\`, \`handlers/\`) — пошаговый опрос, с учётом что
   пользователь может прислать невалидные данные — переспрашиваем вежливо, не сбрасывая
   уже введённые корректные поля.
5. **crm_client.py — retry-логика.** Реализовать асинхронный POST с ретраями строго по
   числам из раздела 2. Не блокировать event loop во время ожидания (\`asyncio.sleep\`,
   не \`time.sleep\`). Написать тест на respx, эмулирующий 500→500→200 и 429×3→fail,
   ДО того как гонять это руками через реальный Beeceptor — реальный сервис недетерминирован
   (то самое "каждый 5-й запрос"), для юнит-теста нужен мок.
6. **Docker.** Dockerfile (multi-stage не обязателен для такого размера, но слой
   зависимостей отдели от слоя кода для кэша), docker-compose.yml с \`env_file: .env\`,

7. **End-to-end проверка** с реальным Beeceptor-эндпоинтом (после того как владелец
   подтвердил, что он настроен) — раздел 6.
8. **README + DEVLOG** — оформляются по ходу работы, не одним махом в конце (см. раздел 9,

## 6. ВАЛИДАЦИЯ И ВЕЖЛИВЫЙ ИНТЕРФЕЙС — ТОЧНЫЕ ТРЕБОВАНИЯ

**Имя:** 2–50 символов после \`.strip()\`. Не требуй "только буквы" сверх ТЗ (двойные
имена, дефис, апостроф — не блокировать без явной причины). При ошибке — конкретика:
что не так, а не общее "неверный формат".

**Телефон:** должен принимать разумные вариации российского номера:
\`+7 (XXX) XXX-XX-XX\`, \`+7XXXXXXXXXX\`, \`8 XXX XXX-XX-XX\`, с пробелами/без — нормализуй
перед сохранением в единый формат для отправки в CRM. Regex должен быть один, в
\`validators/phone.py\`, с комментарием-примером валидных/невалидных значений — не
разбросан по хендлерам.

**Время связи:** только через кнопки (inline или reply — на выбор, но не текстовый
ввод) — три варианта: Утро, День, Вечер. Исключает опечатки и невалидный текст by design.

**Тон сообщений (обязательно "Вы", без канцелярита):**
- Приветствие — тепло, коротко объясняет, зачем нужен опрос.
- Ошибка валидации — по-доброму объясняет, что поправить, без "Ошибка!" капслоком.
- Ожидание ответа CRM — не молчать: короткое "Отправляю заявку..." перед долгим ожиданием.
- Ретрай — ТОЧНАЯ формулировка из ТЗ: «Сервер CRM временно занят, пытаюсь отправить
  еще раз...» (можно чуть варьировать по попыткам, но не терять смысл).
- Финальный сбой (после 3 попыток) — извинение + понятное объяснение (не "Error 500"),
  без создания паники; по возможности — без потери введённых пользователем данных
  (кнопка "Попробовать снова" на карточке, не пересбор с нуля).

## 8. ЧТО СЧИТАЕТСЯ "МУСОРОМ ОТ ИИ" — ЗАПРЕЩЕНО ЯВНО

Это отдельно оцениваемый критерий, не абстрактное пожелание "пиши чистый код":

- Закомментированный код ("на всякий случай оставил") — удалять, не комментировать.
- \`# TODO\`, \`# FIXME\`, \`pass  # implement later\` — задача либо доделана, либо не входит
  в скоуп и не упоминается вовсе.
- Docstring, дублирующий сигнатуру ("Function to validate phone number that validates
  the phone number") — либо содержательный docstring (что считается валидным, почему
  regex такой), либо никакого.
- Неиспользуемые импорты, неиспользуемые переменные, неиспользуемые параметры функций.
- \`try/except Exception: pass\` — заглатывание ошибок без логирования и без причины.
- Дублирование одной и той же логики валидации/форматирования в нескольких местах
  вместо вызова общей функции.
- Абстракции "на будущее" без текущей необходимости (базовые классы под одну
  реализацию, конфигурируемые фабрики под один сценарий, ORM под задачу, где хватает
  FSM-хранилища).
- Несогласованный стиль наименований в разных файлах (где-то \`snake_case\`, где-то
  \`camelCase\` в Python) — весь код проходит \`ruff\`/\`black\` (или эквивалент для
  выбранного стека) перед сдачей.
- Магические числа, продублированные в нескольких местах (3 попытки, 3 секунды,
  2–50 символов) — только в \`config.py\`/константах, один раз.

Перед финальной сдачей — самопроверка по этому списку как явный шаг, не полагаясь на
то, что "код и так получился чистым".

  \`\`\`
- Фиксируй **реальные** развилки, а не выдумывай задним числом: например, почему
  aiogram 3.x, а не 2.x или \`python-telegram-bot\`; почему \`httpx\`, а не \`aiohttp\` для

- Не приукрашивай и не сглаживай неудачные попытки — итерация "сделал → не сработало →
